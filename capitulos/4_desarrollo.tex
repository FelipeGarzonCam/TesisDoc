\chapter{DESARROLLO}

Este capítulo detalla el proceso de diseño, construcción e implementación de la aplicación web \textit{QuickTable} para la gestión de comandas y su respectivo módulo de seguridad por hardware. El desarrollo se ejecutó siguiendo las fases metodológicas planteadas, abarcando desde la definición de requerimientos hasta las pruebas finales del sistema.

\section{Levantamiento y Especificación de Requerimientos}

De acuerdo con la metodología establecida, la primera fase del desarrollo consistió en la especificación formal de los requerimientos del software. Para garantizar la trazabilidad y la calidad en el diseño, se tomó como marco de referencia el estándar ISO/IEC/IEEE 29148 \cite{ISO29148:2018}, el cual proporciona lineamientos para la ingeniería de requerimientos en sistemas de software.

Se definieron requerimientos orientados a solucionar las problemáticas operativas, sanitarias y de seguridad informática identificadas en el sector gastronómico. A partir de la arquitectura cliente-servidor propuesta, se establecieron roles específicos para garantizar el principio del menor privilegio y segmentar las responsabilidades del negocio.


\subsection{Casos de Uso y Roles del Sistema}

El sistema fue diseñado para soportar múltiples actores operando de manera concurrente en una red local (intranet). Se definieron cinco roles principales, los cuales interactúan de forma diferenciada con el sistema:

\begin{itemize}
	\item \textbf{Administrador:} Encargado de gestionar el menú, los empleados, las mesas y visualizar reportes de ventas. Tiene acceso completo al sistema mediante validación de doble factor (2FA).
	\item \textbf{TI (Tecnología de la Información):} Rol con privilegios exclusivos para la gestión de seguridad, como la encriptación de la base de datos y la asignación de llaves físicas (NFC) a los administradores.
	\item \textbf{Mesero:} Responsable de la creación de nuevas comandas, adición de productos a pedidos existentes y visualización del estado de preparación de los platos.
	\item \textbf{Cocina:} Rol operativo que visualiza en tiempo real los pedidos entrantes y actualiza su estado (Pendiente, En Preparación, Listo).
	\item \textbf{Caja:} Encargado de procesar la facturación de las comandas consolidadas, liberar las mesas y cerrar el ciclo de venta.
\end{itemize}

Debido a la cantidad de actores y funcionalidades involucradas, el diagrama general de casos de uso se descompuso en tres vistas complementarias: autenticación y asistencia, operación diaria y administración del sistema. Cada una de estas vistas agrupa los casos de uso según el tipo de interacción y el área funcional del restaurante, lo que facilita su lectura y trazabilidad respecto a los módulos implementados en la aplicación web.

\subsection{Requerimientos No Funcionales}

Además de las funcionalidades, el diseño del sistema debió contemplar restricciones técnicas que garantizaran su viabilidad operativa, seguridad, mantenibilidad y bajo costo de implementación, considerando las capacidades tecnológicas habituales de los restaurantes PYME.

\begin{itemize}
	\item \textbf{Multiplataforma:} La interfaz de usuario debe ser accesible y responsiva desde dispositivos móviles (smartphones y tablets) y computadores de escritorio mediante navegadores web modernos, evitando la necesidad de instalar aplicaciones nativas.
	
	\item \textbf{Despliegue Local (Intranet):} El sistema debe operar sobre una red local del establecimiento, sin depender de una conexión permanente a internet para ejecutar sus funciones principales.
	
	\item \textbf{Tiempos de Respuesta:} Las operaciones de consulta, registro y actualización de comandas deben ejecutarse con tiempos de respuesta suficientemente bajos para no afectar el flujo operativo del restaurante, idealmente en menos de 2 segundos por transacción.
	
	\item \textbf{Optimización de Recursos:} El backend y la base de datos deben poder ejecutarse sobre hardware de gama de entrada, con el fin de mantener bajos los costos de adopción para pequeñas y medianas empresas.
	
	\item \textbf{Seguridad de Sesión:} El sistema debe administrar sesiones de usuario con mecanismos que reduzcan accesos no autorizados, incluyendo expiración por inactividad y uso de cookies seguras de sesión.
	
	\item \textbf{Protección de Datos Sensibles:} La información sensible, como credenciales y datos asociados a mecanismos de autenticación, debe almacenarse utilizando técnicas de protección criptográfica apropiadas.
	
	\item \textbf{Respaldo y Recuperación:} El sistema debe permitir la generación y restauración de copias de seguridad para proteger la información operativa frente a fallos del sistema o pérdida de datos.
	
	\item \textbf{Mantenibilidad:} La solución debe conservar una estructura modular que facilite la corrección de errores, la evolución del sistema y la incorporación de nuevas funcionalidades sin afectar los componentes ya implementados.
	
	\item \textbf{Trazabilidad Operativa:} El sistema debe registrar eventos relevantes de acceso y cierre de sesión para facilitar el control administrativo y el seguimiento de la operación diaria.
\end{itemize}

\subsection{Arquitectura General del Sistema}
Para el desarrollo de la aplicación se implementó una arquitectura cliente-servidor basada en el modelo de tres capas: Presentación, Lógica de Negocio y Datos. Esta estructura garantiza la separación de responsabilidades, facilitando la escalabilidad, el mantenimiento del código y la seguridad del sistema. El patrón arquitectónico principal utilizado fue el Modelo-Vista-Controlador (MVC), adaptado mediante la tecnología Razor Pages de .NET 8, la cual encapsula el modelo y el controlador en una clase unificada (\texttt{PageModel}) vinculada directamente a su respectiva vista, optimizando el flujo de trabajo en aplicaciones orientadas a páginas.

La arquitectura general se compone de los siguientes elementos:

\begin{itemize}
	\item \textbf{Capa de Presentación (Frontend):} Desarrollada utilizando HTML5, CSS3, JavaScript y el framework Bootstrap. La interfaz de usuario toma como base estructural la plantilla AdminLTE, la cual fue extendida y fuertemente personalizada para adaptar la apariencia a las necesidades visuales y operativas del sistema en dispositivos móviles y de escritorio.
	\item \textbf{Capa de Aplicación (Backend):} Construida con el framework .NET 8 en el lenguaje C\#. Es responsable de procesar las peticiones HTTP, ejecutar las reglas de negocio y gestionar la comunicación con el módulo de hardware RFID mediante Endpoints de API tipo REST.
	\item \textbf{Capa de Datos:} Implementada sobre el motor de base de datos relacional Microsoft SQL Server, utilizando el ORM Entity Framework 6 (EF6) para garantizar una comunicación segura y transaccional.
	\item \textbf{Módulo de Seguridad (Hardware):} Compuesto por una Raspberry Pi interactuando con un lector NFC RFID-RC522, el cual actúa como un cliente independiente para la validación física del administrador en el esquema de doble factor de autenticación (2FA).
\end{itemize}

\section{Diseño e Implementación de la Base de Datos}

El diseño de la base de datos se estructuró para soportar las operaciones críticas del restaurante de manera transaccional, garantizando la integridad referencial y minimizando la redundancia de datos. Se optó por el motor Microsoft SQL Server debido a su robustez, soporte nativo de encriptación y excelente integración con el ecosistema de desarrollo de Microsoft.Para proteger los archivos físicos de la base de datos en reposo, se habilitó el mecanismo de \textit{Transparent Data Encryption} (TDE) nativo de SQL Server. 
Este mecanismo cifra los archivos \texttt{.mdf} y \texttt{.ldf} directamente en el 
disco, de forma transparente para la aplicación, utilizando una clave de cifrado de 
base de datos protegida por un certificado de servidor. Los archivos de certificado 
(\texttt{.cer}) y de clave privada (\texttt{.pvk}) generados son almacenados de forma 
segura por el rol TI, garantizando que cualquier copia de seguridad del sistema sea 
ilegible sin la posesión de dichas credenciales criptográficas.

\subsection{Modelo Entidad-Relación}

El esquema de la base de datos consta de tablas interrelacionadas que representan las entidades fundamentales del negocio y los mecanismos de seguridad. Entre las entidades principales se encuentran:

\begin{itemize}
	\item \textbf{Empleados:} Almacena las credenciales protegidas (mediante hashing BCrypt) de los usuarios, sus datos personales, el UID asociado (si aplica) y el rol asignado dentro del sistema (Admin, Mesero, Cocina, Cajero).
	\item \textbf{MenuItems (Menú):} Contiene el catálogo de productos disponibles, sus descripciones, categorías y precios, permitiendo la construcción dinámica de los pedidos.
	\item \textbf{PedidosActivos e ItemDetalle:} Tablas transaccionales encargadas de almacenar el encabezado de la comanda en curso (fecha, mesero, mesa, totales y estado de preparación) y los ítems individuales con sus cantidades y observaciones, respectivamente.
	\item \textbf{HistorialPedidos e HistorialDetalle:} Entidades destinadas a la persistencia de las comandas ya facturadas y finalizadas, garantizando la inmutabilidad de los datos históricos de ventas y facturación.
	\item \textbf{TarjetasRC:} Tabla dedicada a la seguridad por hardware, que almacena el UID físico de las tarjetas NFC y gestiona la asignación activa a los usuarios administradores.
	\item \textbf{RegistroSesiones y Codigos2FA:} Entidades de auditoría y control de acceso. La primera registra las horas de conexión y desconexión de los empleados para el control de asistencia, mientras que la segunda gestiona de forma temporal los tokens para la autenticación de doble factor.
\end{itemize}

Como se observa en la Figura \ref{fig:diagrama_mer}, el modelo de datos fue diseñado siguiendo un enfoque relacional normalizado que separa claramente la operación en tiempo real de la persistencia histórica. La entidad central del sistema es \textbf{Empleado}, la cual establece una relación de 'uno a muchos' tanto con los \textbf{PedidosActivos} como con el \textbf{RegistroSesion}, permitiendo la trazabilidad de qué usuario (Mesero) crea cada comanda y en qué horarios labora el personal. Asimismo, mantiene una relación opcional de 'uno a uno' con la tabla \textbf{TarjetaRC}, utilizada exclusivamente para vincular credenciales físicas NFC a los perfiles con rol de Administrador.

El flujo transaccional del restaurante se maneja a través de dos estructuras gemelas: una para la operación en curso y otra para el archivo histórico. Un \textbf{PedidoActivo} actúa como entidad cabecera que se relaciona de 'uno a muchos' con \textbf{ItemDetalle}, donde cada detalle hace referencia mediante llave foránea al producto específico en la tabla \textbf{MenuItem}. Una vez que el Cajero procesa el pago y finaliza la orden, la integridad de los datos se garantiza migrando la información hacia las entidades \textbf{HistorialPedido} e \textbf{HistorialDetalle}. Esta separación arquitectónica evita la sobrecarga de la tabla de pedidos activos, garantizando tiempos de respuesta rápidos (menores a 2 segundos) para el personal de Meseros y Cocina, mientras se conservan los datos inmutables de facturación para consultas administrativas.

\subsection{Implementación con Entity Framework 6}
La arquitectura de persistencia de datos del sistema fue implementada utilizando Entity Framework 6 (EF6) bajo el ecosistema de .NET 8. Se optó por EF6 debido a su alta madurez y estabilidad a nivel empresarial. Al contar con una extensa documentación y un motor de persistencia rigurosamente probado contra fallos transaccionales, se reduce la curva de aprendizaje para futuros mantenimientos, alineándose con el objetivo de ofrecer una solución escalable y de bajo costo para las PYMES.

Para gestionar la interacción con la base de datos, el sistema define un contexto principal denominado \texttt{SistemaQuickTableContext}, el cual hereda de la clase base \texttt{DbContext}. La persistencia se estructura bajo los siguientes principios:

\begin{itemize}
	\item \textbf{Mapeo Objeto-Relacional Consolidado:} Las tablas de SQL Server se representan directamente mediante entidades (clases) ubicadas en el directorio \texttt{Dominio} (ej. \texttt{PedidoActivo}, \texttt{ItemDetalle}). El contexto administra el estado de estas entidades a través de colecciones \texttt{DbSet<T>}, garantizando que las modificaciones en la lógica de negocio se traduzcan de manera íntegra a la base de datos.
	\item \textbf{Gestión Segura de Conexiones:} La cadena de conexión al motor SQL Server se inyecta dinámicamente desde el archivo de configuración \texttt{appsettings.json}. Esta separación aísla las credenciales de acceso del código fuente principal, cumpliendo con las buenas prácticas de seguridad informática.
	\item \textbf{Abstracción de Consultas con LINQ:} Las operaciones CRUD ejecutadas por los diferentes módulos no emplean sentencias SQL estáticas. En su lugar, se utiliza \textit{Language Integrated Query} (LINQ), lo cual mitiga por diseño las vulnerabilidades de inyección SQL y optimiza la legibilidad del código al expresar reglas de negocio en lenguaje nativo C\#.
\end{itemize}


\section{Implementación de la Solución Web}

El núcleo central del sistema fue desarrollado como una aplicación web robusta, diseñada para centralizar la comunicación entre las áreas del restaurante y eliminar las ineficiencias del papel. El desarrollo se fundamentó en el ecosistema de Microsoft, utilizando .NET 8 y el patrón arquitectónico MVC (Modelo-Vista-Controlador) adaptado mediante el enfoque de \textit{Razor Pages}.

\subsection{Interfaz de Usuario (Frontend y Personalización visual)}

Como se estableció en el diseño arquitectónico, la capa de presentación toma como base estructural el framework \textbf{AdminLTE} (soportado por Bootstrap). Sin embargo, su implementación no se limitó al uso genérico de la plantilla; esta fue sometida a una \textbf{reestructuración y personalización profunda mediante hojas de estilo propias (CSS)}. El objetivo de esta sobreescritura de estilos fue transformar un panel de control administrativo estándar en una interfaz operativa ágil, adaptada específicamente a las condiciones físicas del sector gastronómico.

Esta implementación personalizada aportó las siguientes características técnicas al proyecto:

\begin{itemize}
	\item \textbf{Diseño Responsivo y Optimización Táctil:} Si bien la estructura de grillas de Bootstrap garantiza la adaptabilidad a diferentes resoluciones, el CSS personalizado se enfocó en ampliar las áreas de interacción (botones, selectores de cantidad y tarjetas de menú). Esto es vital para facilitar la precisión táctil en los \textit{smartphones} de los meseros y evitar toques accidentales durante operaciones rápidas.
	\item \textbf{Minimalismo y Contraste Visual:} Se sobrescribieron los estilos predeterminados de AdminLTE para ocultar elementos modulares innecesarios (como menús laterales complejos) en las vistas operativas. Asimismo, se implementó una paleta de colores de alto contraste que permite a los cocineros y cajeros identificar visualmente y a distancia el estado de un pedido (por ejemplo, alertas visuales para comandas en espera prolongada o listas para entrega).
	\item \textbf{Interactividad Asíncrona:} Para complementar el diseño visual, se integró JavaScript (junto con librerías como jQuery y DataTables) en las vistas de Razor Pages. Esto permitió habilitar acciones dinámicas sin necesidad de recargar la página completa, optimizando tiempos muertos al momento de filtrar el menú o actualizar el estado de preparación de un plato.
\end{itemize}


\subsection{Lógica de Negocio y Módulos .NET 8 (Razor Pages)}
El backend de la aplicación gestiona la lógica de negocio a través del enfoque de Razor Pages, el cual agrupa los controladores y las vistas en un diseño orientado a páginas (\textit{Page-Focused}), resultando en un código más organizado y fácil de mantener que el MVC tradicional para este tipo de aplicaciones. 

El sistema fue segmentado en cinco módulos principales, protegidos por un sistema de autorización basado en reclamaciones (\textit{Claims}) que restringe el acceso según el rol del empleado autenticado:

\begin{itemize}
	\item \textbf{Módulo de Meseros:} Implementa el modelo \texttt{PageModel} para recibir peticiones HTTP POST con objetos JSON que contienen los arreglos de productos. El controlador procesa la lógica matemática de impuestos y subtotales en el servidor y ejecuta la persistencia inicial del registro en la tabla \texttt{PedidosActivos}.
	\item \textbf{Módulo de Cocina:} Funciona mediante consultas asíncronas periódicas (AJAX) que interrogan a la base de datos para extraer dinámicamente los registros con estado pendiente. A través de Endpoints específicos, permite la actualización de estados y marcas temporales sin necesidad de recargar el DOM completo.
	\item \textbf{Módulo de Caja:} Su controlador gestiona el ciclo de vida final del objeto pedido. Tras la confirmación del pago mediante peticiones asíncronas, el sistema ejecuta la lógica transaccional de migración de datos, trasladando la información desde las tablas operativas (\texttt{PedidosActivos}) hacia las entidades inmutables del esquema (\texttt{HistorialPedidos}).
	\item \textbf{Módulos de Administración y TI:} Exponen las interfaces de configuración del CRUD de inventarios y empleados, y gestionan los algoritmos criptográficos (hashing de credenciales). Además, orquestan la comunicación externa exponiendo los Endpoints REST de la API que interactúan con la Raspberry Pi para la verificación de tarjetas NFC.La gestión criptográfica del sistema se estructura en dos servicios independientes. 
	El \texttt{PasswordService} implementa el algoritmo de hashing \textit{BCrypt} con 
	factor de trabajo 12, lo que garantiza que las contraseñas nunca se almacenen en 
	texto plano y que el costo computacional de verificación sea suficiente para resistir 
	ataques de fuerza bruta por diccionario. El \texttt{CryptoService} implementa el 
	estándar de cifrado simétrico \textit{AES} (Advanced Encryption Standard) en modo 
	CBC (\textit{Cipher Block Chaining}) con clave de 256 bits, utilizado específicamente 
	para cifrar los identificadores UID de las tarjetas NFC antes de su almacenamiento 
	en la tabla \texttt{TarjetasRC}, asegurando que un acceso no autorizado a la base de 
	datos no exponga los identificadores físicos de las llaves de acceso administrativo.
\end{itemize}

\section{Implementación del Módulo de Seguridad por Hardware (2FA Local)}

En el contexto del desarrollo de aplicaciones web contemporáneas, la implementación de mecanismos de Autenticación de Doble Factor (2FA) se ha estandarizado mediante el uso de servicios basados en la nube, tales como el envío de tokens por SMS, correos electrónicos o aplicaciones generadoras de códigos (ej. Microsoft Authenticator). Sin embargo, en el marco de este proyecto, dichos métodos presentan un obstáculo técnico restrictivo: \textbf{su dependencia de una conexión activa a Internet}.

Dado que uno de los requerimientos no funcionales prioritarios del sistema es garantizar su funcionamiento ininterrumpido sobre una red local (Intranet) con el fin de reducir los costos operativos mensuales para el establecimiento, se descartó la integración de sistemas 2FA en la nube. En su lugar, se diseñó e implementó un módulo de seguridad por hardware basado en tecnología NFC (\textit{Near Field Communication}). Este enfoque añade una capa de seguridad basada en un factor físico de posesión ('algo que el usuario tiene'), garantizando la presencia in situ del administrador al momento de autorizar accesos críticos, operando de manera 100\% \textit{offline} e integrándose de forma nativa con el servidor .NET 8.

\subsection{Arquitectura y Configuración del Hardware}

El dispositivo de validación física actúa como un cliente independiente dentro de la red local del restaurante. Su arquitectura de hardware se compone de dos elementos principales:

\begin{itemize}
	\item \textbf{Unidad de Procesamiento:} Se utilizó una placa \textit{Raspberry Pi 3 Model B v1.2}, la cual ofrece conectividad de red local (Wi-Fi/Ethernet) y los pines de propósito general (GPIO) necesarios para la interacción con periféricos externos.
	\item \textbf{Lector RFID/NFC:} Se integró el módulo \textit{RFID-RC522}, que opera a una frecuencia de 13.56 MHz. Este sensor fue seleccionado por su bajo costo, bajo consumo energético y alta fiabilidad en entornos comerciales.
\end{itemize}

La comunicación entre la Raspberry Pi y el lector RC522 se estableció mediante el protocolo de comunicación síncrona \textbf{SPI} (\textit{Serial Peripheral Interface}). Se configuraron los pines físicos correspondientes (MOSI, MISO, SCK, SDA y RST) para garantizar una transmisión de datos rápida y libre de interferencias al momento de leer el identificador único (UID) de las tarjetas asignadas a los administradores.

\subsection{Desarrollo del Cliente Python e Interfaz Gráfica (Tkinter)}
Para gobernar el comportamiento del hardware de validación física, se desarrolló un script en Python 3 que se ejecuta localmente en la Raspberry Pi. Este lenguaje fue seleccionado por su alta compatibilidad con el sistema operativo (Raspberry Pi OS) y la madurez de sus librerías para la manipulación de los puertos GPIO.

Para optimizar la interacción del usuario sin requerir periféricos externos como teclados o ratones, el dispositivo fue dotado de una Interfaz Gráfica de Usuario (GUI) orientada a pantallas táctiles, desarrollada con la librería estándar \texttt{Tkinter}. El diseño presenta un teclado numérico renderizado en pantalla (\textit{numpad}), permitiendo al administrador digitar un código de sesión dinámico temporal que vincula la lectura física de la tarjeta NFC con el intento de inicio de sesión en el navegador web.

El funcionamiento interno del programa se apoya en la integración concurrente de tres librerías fundamentales:
\begin{enumerate}
	\item \textbf{Tkinter (GUI):} Encargada de la construcción visual y de capturar las pulsaciones del usuario en la pantalla táctil mediante su bucle principal de eventos (\textit{mainloop}).
	\item \textbf{mfrc522 (Hardware):} Librería especializada que inicializa la comunicación por el bus SPI, activa el campo electromagnético del sensor RC522 y extrae el identificador único (UID) crudo de las tarjetas NFC.
	\item \textbf{Requests (Red):} Biblioteca HTTP de alto nivel utilizada para empaquetar el UID leído y el código de sesión en formato JSON, formulando una petición HTTP \texttt{POST} asíncrona hacia el Endpoint REST del servidor .NET 8 (ej. \texttt{/api/auth/validate-nfc}).
\end{enumerate}

Uno de los principales retos técnicos en este script fue evitar el bloqueo (\textit{freezing}) de la interfaz gráfica durante el proceso de lectura. Dado que la función de lectura de la librería \texttt{mfrc522} es bloqueante (espera indefinidamente hasta detectar una tarjeta), ejecutarla en el hilo principal detendría la pantalla. Para solucionar esto, se implementó un modelo de programación concurrente mediante \textit{Threading}. Al confirmar el código de sesión, la GUI delega la lectura NFC y la transmisión HTTP a un hilo secundario (\textit{Worker Thread}). Así, la interfaz sigue respondiendo y mostrando retroalimentación visual ("Esperando tarjeta...") hasta que el servidor retorna un código de estado HTTP (200 OK o 401 Unauthorized), concluyendo el ciclo de comunicación.


\subsection{Sincronización y Criptografía con el Backend (.NET 8)}

El éxito operativo de este módulo radica en su perfecta sincronización con la capa de aplicación web, diseñada bajo estrictos controles de seguridad perimetral. 

Cuando el UID de la tarjeta es recibido por el servidor a través del \textit{Endpoint} de la API, el sistema .NET 8 no lo procesa ni lo almacena en texto plano. El backend recupera el UID encriptado de la base de datos (protegido previamente mediante algoritmos de cifrado simétrico) asociado al perfil del administrador en cuestión. Si la validación criptográfica es exitosa y el código de sesión coincide con una solicitud web activa y no expirada, el servidor actualiza el estado de los reclamos (\textit{Claims}) de autenticación del usuario a 'Autorizado'.

De manera simultánea, la vista del navegador en la estación de trabajo realiza consultas asíncronas periódicas mediante JavaScript (método de \textit{polling}) para verificar su estado. Al detectar la autorización emitida por la Raspberry Pi, el sistema redirige automáticamente al administrador a su panel de control privado, completando el ciclo de validación sin necesidad de exponer credenciales o tokens a redes externas.

La sincronización entre el cliente Python y el backend \texttt{.NET~8} se basa en 
el intercambio de tokens de sesión de corta duración gestionados por la tabla 
\texttt{Codigos2FA}. El UID físico leído por el lector RFID-RC522 nunca se transmite 
en texto plano a través de la red local; en su lugar, el backend valida la sesión 
mediante el token y confronta internamente el identificador cifrado almacenado en 
la tabla \texttt{TarjetasRC} utilizando el \texttt{CryptoService} basado en 
AES-256-CBC con clave de 256~bits. Este diseño garantiza que, incluso ante una 
intercepción del tráfico en la intranet, un atacante no pueda obtener el 
identificador real de la tarjeta NFC ni suplantar el segundo factor de autenticación.

\section{Pruebas de Validación y Verificación}

\end{itemize}

\subsection{Pruebas Unitarias y de Seguridad Lógica}

Para certificar la robustez interna de la aplicación, especialmente en el manejo de datos sensibles y en los mecanismos criptográficos implementados en el sistema, se incorporó una batería de pruebas a nivel de código utilizando el marco de pruebas \textit{MSTest} para .NET 8.

A diferencia de las pruebas de interfaz, estas evaluaciones se enfocaron en validar de forma aislada la lógica de negocio y los componentes de seguridad implementados en el \textit{backend}. Los escenarios evaluados incluyeron:
\begin{itemize}
	\item \textbf{Cifrado y Descifrado de Identificadores NFC:} Validación del servicio criptográfico encargado de proteger los identificadores únicos de las tarjetas NFC utilizadas en el segundo factor de autenticación, comprobando que el proceso de cifrado no altere la posibilidad de recuperación íntegra del valor original.
	\item \textbf{Manejo de Entradas Inválidas:} Verificación del comportamiento del servicio criptográfico frente a valores nulos, vacíos o estructuralmente inválidos, garantizando que el sistema responda de forma segura sin comprometer la estabilidad del \textit{backend}.
	\item \textbf{Hashing y Verificación de Contraseñas:} Validación del servicio responsable de generar y comprobar contraseñas cifradas, asegurando que las credenciales no se almacenen en texto plano y que la verificación funcione correctamente tanto para credenciales válidas como incorrectas.
\end{itemize}
Las pruebas unitarias cubren específicamente los dos servicios criptográficos 
del sistema. Para el \texttt{CryptoService}, los casos de prueba verifican que 
el ciclo de cifrado y descifrado con AES-256-CBC preserve la integridad del dato 
original, y que entradas inválidas sean rechazadas sin propagar excepciones al 
resto del sistema. Para el \texttt{PasswordService}, se valida que el hashing 
con BCrypt (factor de trabajo~12) produzca resultados no reversibles y que el 
mecanismo de verificación opere correctamente tanto para credenciales válidas 
como inválidas, cubriendo los requerimientos RS-01 y RS-02 de la 
Tabla~\ref{tab:req_funcionales_seguridad}.

\subsection{Pruebas de API con Postman}

Con el fin de complementar las pruebas unitarias y verificar el comportamiento de los servicios HTTP expuestos por la aplicación, se realizaron pruebas manuales de API mediante la herramienta \textit{Postman}. Esta fase permitió validar directamente la estructura de las respuestas del \textit{backend}, sin depender de la interfaz gráfica del sistema.

Las pruebas realizadas se concentraron principalmente en los servicios administrativos y operativos ya implementados en la solución. Entre los endpoints validados se incluyeron la creación de copias de seguridad, la consulta del historial de respaldos, la verificación del espacio disponible para almacenamiento y el registro de salida de personal. Esta etapa permitió comprobar que los controladores respondieran con objetos JSON válidos, con los campos esperados y con semántica coherente frente a operaciones exitosas o fallidas.

Desde el punto de vista metodológico, el uso de Postman resultó especialmente útil para confirmar el contrato de comunicación entre cliente y servidor, verificar parámetros de entrada, revisar estructuras de respuesta y documentar evidencia visual de los resultados obtenidos. De esta manera, las pruebas de API constituyen un nivel intermedio de validación entre las pruebas unitarias del código y las pruebas funcionales completas sobre la interfaz del sistema.

\subsection{Evaluación del Tiempo de Respuesta Operativa}

En estricta correspondencia con el objetivo de erradicar los retrasos asociados al modelo manual de toma de pedidos, se diseñó una etapa de medición empírica de la latencia del sistema. Para garantizar que la evaluación reflejara fielmente las condiciones reales de implementación en un restaurante PYME, el registro de tiempos se estructuró en dos escenarios de red diferenciados: una ejecución en un entorno de intranet aislada operando exclusivamente sobre la red de área local inalámbrica (WLAN) y cableada (LAN), y una ejecución conectada a internet, con el fin de verificar si el tráfico externo introduce algún grado de interferencia o sobrecarga en la latencia de los componentes locales.

El registro de tiempos se focalizó en las operaciones transaccionales más demandantes, utilizando tanto herramientas de diagnóstico integradas en el navegador (\textit{DevTools - Network}) como cronometría física. Las variables críticas objeto de medición fueron:

\begin{itemize}
	\item \textbf{Latencia de propagación de comandas:} el tiempo neto transcurrido desde que el agente Mesero (conectado por Wi-Fi) confirma el envío de un pedido, hasta que los datos son persistidos en SQL Server y la nueva orden es renderizada correctamente en la pantalla de la estación de Cocina (conectada por cable).
	\item \textbf{Rendimiento de consultas complejas:} la latencia experimentada en la resolución de operaciones financieras pesadas, tales como la generación de historiales de pedidos y reportes de facturación consolidados en el módulo del Administrador.
	\item \textbf{Latencia del flujo de Autenticación de Doble Factor (2FA):} el tiempo de procesamiento integral del hardware de seguridad, abarcando desde el instante físico en que la tarjeta NFC hace contacto con el lector RFID-RC522 conectado a la Raspberry Pi, pasando por la validación criptográfica del cliente Python contra la API .NET, hasta que el sistema web autoriza finalmente el inicio de sesión del usuario en pantalla.
\end{itemize}

El criterio de aceptación ineludible para esta fase de pruebas dicta que, incluso operando sobre los equipos de hardware heredado descritos en la arquitectura del sistema e independientemente del escenario de red (con o sin salida a internet), el tiempo de respuesta total para cualquiera de estas operaciones críticas no debe exceder el umbral máximo de \textbf{2,0 segundos}.

\subsection{Pruebas Automatizadas de Concurrencia}

Para superar las limitaciones de una validación manual y esporádica, y garantizar la estabilidad del sistema ante cargas de trabajo sostenidas durante una jornada operativa completa, se diseñó un entorno de estrés basado en agentes virtuales autónomos desarrollados con \textit{Playwright} sobre Node.js. A diferencia de las herramientas de prueba de carga convencionales, Playwright instancia navegadores Chromium reales que interactúan con la interfaz gráfica exactamente como lo haría un usuario humano: navegan, hacen clic, llenan formularios y procesan respuestas del servidor, lo cual permite detectar no solo errores de \textit{backend} sino también fallos de renderizado, condiciones de carrera en el \textit{frontend} y regresiones visuales.

\subsubsection*{Arquitectura del entorno de automatización}

El sistema se estructuró en dos proyectos independientes de Playwright escritos en TypeScript, desplegados sobre tres equipos físicos disponibles. Dado que la configuración ideal contempla un agente por equipo, la limitación de hardware obligó a distribuir los cuatro agentes de Mesero y los dos agentes de rol fijo entre las máquinas disponibles de la siguiente manera:

\begin{itemize}
	\item \textbf{Equipo 1 (conexión LAN por cable):} ejecutó los agentes de Cocina y Caja de forma paralela.
	\item \textbf{Equipo 2 (conexión Wi-Fi 2.4\,GHz):} ejecutó dos instancias del agente Mesero de forma simultánea, cada una autenticada con credenciales de usuario distintas, replicando la operación de dos meseros atendiendo mesas en movimiento constante a través de la red inalámbrica.
	\item \textbf{Equipo 3 (conexión Wi-Fi 2.4\,GHz):} ejecutó las dos instancias restantes del agente Mesero, también con credenciales independientes, completando el total de cuatro agentes Mesero activos durante la jornada.
\end{itemize}

El agente Mesero se configura con un único trabajador activo por instancia, un tiempo de espera máximo global de 10 horas y un perfil de dispositivo móvil que instruye a Chromium para emular la resolución de pantalla, la densidad de píxeles, la cadena de agente de usuario y el modo táctil propios de un teléfono iPhone 12. Esta elección no es arbitraria: replica fielmente el escenario real donde el mesero opera desde un dispositivo móvil en movimiento constante. En todos los proyectos, cada interacción individual (clic, llenado de campo o navegación de página) dispone de un tiempo de espera máximo de 30 segundos antes de ser catalogada como fallo transaccional. Las trazas de ejecución se conservan únicamente ante fallos, y las capturas de pantalla se generan de forma automática al detectar errores, lo que facilita el diagnóstico posterior sin incrementar la carga de escritura en disco durante la operación normal.

\subsubsection*{Mecanismo de tiempo emulado y ritmo operativo}

Un aspecto central del diseño es la distinción entre el \textit{tiempo de ejecución real} y el \textit{tiempo de jornada simulada}. Los scripts no comprimen el tiempo: la prueba se ejecuta durante 8 horas de reloj real ininterrumpidas. La duración se controla mediante una constante de 8 horas expresada en milisegundos que se evalúa de forma continua contra el tiempo transcurrido desde el inicio, de modo que el ciclo principal termina únicamente cuando han transcurrido exactamente 8 horas de tiempo de pared (\textit{wall-clock time}).

Dentro de ese marco temporal, cada agente modela el comportamiento humano mediante pausas activas que suspenden la corrutina sin bloquear el hilo de ejecución de Node.js:

\begin{itemize}
	\item \textbf{Agente Mesero:} el intervalo entre pedidos se calcula en función de la hora del día simulada, la cual avanza una hora por cada 60 minutos reales transcurridos partiendo de las 11:00 h, de modo que la prueba recorre una jornada operativa completa en tiempo real. En los periodos de alta demanda (mediodía entre las 12:00 y las 14:00 h, y servicio nocturno entre las 19:00 y las 21:00 h) la pausa entre pedidos oscila aleatoriamente entre 2 y 4 minutos, simulando el ritmo acelerado de un restaurante lleno. Fuera de esas franjas, la espera se extiende entre 2 y 5 minutos, representando los periodos de menor afluencia de clientes.
	\item \textbf{Agente Cocina:} revisa la bandeja de pedidos pendientes con una frecuencia aleatoria de entre 1 y 5 minutos, modelando el comportamiento de un cocinero que consulta la pantalla periódicamente entre preparaciones.
	\item \textbf{Agente Caja:} opera con ciclos de revisión de entre 5 y 15 minutos, reflejando la cadencia más espaciada del cajero, quien factura lotes de pedidos una vez que la cocina los ha despachado.
\end{itemize}

Con este diseño, la prueba de 8 horas genera aproximadamente entre 100 y 130 pedidos distribuidos orgánicamente a lo largo de la jornada, con concentraciones visibles en los picos horarios, produciendo una carga asimétrica y realista sobre el servidor en lugar de una tasa de peticiones constante.

\subsubsection*{Flujo de interacción del agente Mesero}

El agente Mesero estructura su operación en tres funciones asíncronas que se suceden en bucle durante las 8 horas. En primer lugar, al inicio de la sesión, el agente navega a la pantalla de autenticación, completa los campos de nombre y contraseña y aguarda la confirmación HTTP del servidor antes de continuar, garantizando que la sesión esté plenamente establecida antes de cualquier operación posterior.

En segundo lugar, antes de generar cada nuevo pedido, el agente inspecciona el objeto JavaScript que la vista del Index del Mesero mantiene en memoria con los pedidos que la cocina ya marcó como listos. Una vez identificados los pedidos pendientes de confirmación, el agente invoca el endpoint correspondiente del \textit{backend} directamente a través de la API de red de Playwright, sin intentar hacer clic sobre los elementos del DOM. Esta decisión de diseño fue necesaria porque la vista del Mesero ejecuta un refresco automático del contenedor de pedidos cada 5 segundos, lo que reconstruye completamente el árbol de elementos de la página. Cualquier referencia a un botón capturada antes del refresco queda huérfana e inutilizable, por lo que la llamada directa al \textit{backend} resultó ser la única estrategia robusta para garantizar la confirmación de entregas sin fallos intermitentes.

En tercer lugar, el agente navega al formulario de nuevo pedido, selecciona aleatoriamente entre 1 y 7 ítems del menú e incrementa las cantidades de cada uno. Adicionalmente, incorpora comentarios escogidos al azar de un conjunto predefinido en el script (por ejemplo, \textit{``Sin cebolla''}, \textit{``Término medio''} o \textit{``Poco picante''}), sin que exista ninguna relación de congruencia entre el comentario seleccionado y el alimento al que se asocia, dado que el propósito es únicamente estresar el campo de texto y verificar su correcta persistencia en la base de datos. Con una probabilidad del 12,5\% por ciclo (equivalente a aproximadamente 1 de cada 8 pedidos),
el agente también ejecuta la función de edición, que selecciona un pedido activo al azar y modifica su contenido, modelando el escenario real en el que un cliente rectifica su orden después de haberla enviado a cocina y validando el flujo de actualización de comandas en caliente.

\subsubsection*{Flujo de interacción del agente Cocina}

El agente de Cocina opera en un ciclo continuo de revisión y despacho de pedidos durante las 8 horas de prueba. Al inicio de cada iteración, el agente espera a que el contenedor principal de pedidos esté presente en la vista antes de intentar cualquier interacción, con un margen de tolerancia de 5 segundos. Si transcurrido ese tiempo el contenedor no responde, el ciclo registra el evento como error y reintenta en el siguiente intervalo.

Una vez confirmada la disponibilidad de la vista, el agente recupera la lista completa de pedidos pendientes y para cada uno extrae dos datos clave directamente desde el texto de los elementos de la interfaz: el número de mesa, obtenido del encabezado principal de la tarjeta, y el identificador numérico del pedido, obtenido del subencabezado. Con esos datos en mano, el agente no intenta localizar ni hacer clic sobre el botón de la interfaz gráfica. En su lugar, ejecuta directamente la llamada AJAX al endpoint del \textit{backend} que marca el pedido como listo, desde el contexto del navegador autenticado, replicando con exactitud la operación que realizaría el cocinero al presionar el botón físicamente.

Esta decisión fue motivada por el mismo problema estructural identificado en el agente Mesero: la vista de Cocina también ejecuta un refresco periódico automático del contenedor de pedidos, lo que reconstruye completamente el DOM a intervalos regulares. Cualquier referencia a un botón capturada antes del refresco queda huérfana e inutilizable, produciendo el error de elemento no adjunto al documento. La invocación directa al \textit{backend} elimina esta fragilidad por completo, ya que no depende en ningún momento del estado del árbol de elementos de la página. Entre el procesamiento de cada pedido, el agente introduce una pausa breve para evitar saturar el servidor con solicitudes en ráfaga consecutiva.

\subsubsection*{Flujo de interacción del agente Caja}

El agente de Caja opera en un ciclo continuo de revisión y facturación de pedidos durante las 8 horas de prueba. Al inicio de cada iteración, el agente navega a su vista de trabajo y espera a que el contenedor principal de pedidos listos esté disponible antes de intentar cualquier interacción. Si no se detectan pedidos listos para cobrar, el ciclo registra el estado y retoma la espera hasta el siguiente intervalo.

Una vez identificados los pedidos disponibles, el agente enfrenta el mismo problema estructural descrito en los agentes de Mesero y Cocina: la vista de Caja también refresca periódicamente su contenedor de tarjetas, reconstruyendo el DOM por completo a intervalos regulares. Por esta razón, el agente no intenta localizar ni hacer clic sobre los botones de la interfaz gráfica para ejecutar la facturación. En su lugar, invoca directamente la función JavaScript de confirmación desde el contexto del navegador autenticado, comunicándose con el \textit{backend} sin depender del estado del árbol de elementos de la página.

Para cada pedido, el agente extrae de la tarjeta correspondiente los datos relevantes de la transacción: el número de mesa, el identificador del pedido y el total a cobrar. A continuación abre el modal de finalización y procede a completar los campos requeridos. El método de pago se selecciona aleatoriamente entre Efectivo, Tarjeta y QR, y el porcentaje de propina se genera como un valor entero aleatorio entre el 0\% y el 15\%, cubriendo así el espectro habitual de comportamiento del consumidor en un restaurante. Para garantizar que el campo de selección del método de pago procese correctamente el cambio de valor, el agente dispara manualmente el evento de modificación sobre el elemento, ya que la interfaz depende de ese evento para actualizar los cálculos del total en pantalla antes de confirmar.

Una vez completados los campos, el agente confirma la operación. La arquitectura de la vista de Caja está diseñada para abrir automáticamente una nueva pestaña del navegador con la factura generada en formato de impresión. Para gestionar este comportamiento sin intervención manual, el agente utiliza un mecanismo de escucha simultánea de eventos que captura la apertura de la nueva pestaña en el mismo instante en que se lanza la confirmación. Tras aguardar la carga completa del documento de factura, el agente cierra la pestaña de forma programática, garantizando que el ciclo de facturación concluya en su totalidad sin dejar instancias de navegador huérfanas que acumulen memoria durante las 8 horas de ejecución. Si por algún motivo el modal no se cierra tras la confirmación (lo que indica un fallo en el procesamiento del \textit{backend}), el agente lo detecta, registra el error con su traza completa en el CSV y fuerza el cierre del modal mediante JavaScript antes de continuar con el siguiente pedido, evitando que un fallo aislado detenga la ejecución de la prueba.

\subsubsection*{Sistema de trazabilidad mediante registros CSV}

Cada agente mantiene y actualiza de forma incremental un archivo de registro en formato CSV durante toda la ejecución. Cada evento se almacena con una marca de tiempo en formato ISO 8601 expresada en UTC, lo que garantiza la correlación precisa entre los registros de los tres agentes incluso ante diferencias de zona horaria entre los equipos cliente. Las acciones registradas se clasifican en tres categorías semánticas:

\begin{itemize}
	\item \textbf{Eventos de sistema:} inicio de jornada, fin de jornada e inicio de sesión exitoso, que delimitan el ciclo de vida de la prueba.
	\item \textbf{Operaciones transaccionales:} creación de pedido, edición de pedido, confirmación de entrega, marcado como listo y facturación, que representan actividad funcional del sistema.
	\item \textbf{Errores:} cualquier evento clasificado como fallo, que registra adicionalmente la traza de la excepción en el campo de detalle para facilitar el diagnóstico posterior.
\end{itemize}

Los esquemas de columnas difieren por rol: el registro del Mesero captura identificador, mesa y detalle del pedido; el de Cocina registra el identificador del pedido y la mesa atendida; y el de Caja extiende el esquema con el método de pago, el porcentaje de propina y el total facturado, almacenando la información financiera de cada transacción para su posterior análisis.

\subsubsection*{Script de consolidación y análisis de resultados}

Una vez concluida la ejecución de los agentes, los cuatro archivos de registro generados son procesados por un script desarrollado en Python con las bibliotecas \textit{pandas} y \textit{matplotlib}. El script sigue un flujo de análisis estructurado en cuatro etapas.

\paragraph{Etapa 1 — Carga y normalización.} El script infiere el tipo de agente a partir del nombre de cada archivo y aplica el esquema de columnas correspondiente. Para gestionar las celdas que contienen comas dentro del campo de detalle (habitual en la lista de ítems del pedido) une con un separador las particiones que excedan el número de columnas fijas esperadas. A continuación, convierte la columna de marca de tiempo al tipo fecha-hora con zona horaria UTC, descarta filas con marcas inválidas y clasifica cada registro en tres categorías booleanas: error, evento de sistema u operación transaccional. Finalmente, concatena todos los conjuntos de datos en un único registro maestro ordenado cronológicamente.

\paragraph{Etapa 2 — Cálculo de métricas.} Para cada agente se calculan el total de registros, la cantidad de operaciones exitosas, la cantidad de errores y la tasa de éxito porcentual, definida como el cociente entre las operaciones exitosas y la suma de operaciones exitosas más errores, expresado en porcentaje. Adicionalmente, se desagrega la actividad por franjas horarias, se clasifican los errores por tipo y rol, y se extraen estadísticas específicas por función: pedidos creados, editados y entregas confirmadas para el Mesero; pedidos marcados como listos para Cocina; y para Caja, el total de transacciones facturadas, el monto total recaudado, la propina promedio y la distribución de métodos de pago.

\paragraph{Etapa 3 — Generación de visualizaciones.} El script produce seis gráficas en formato PNG dentro de la carpeta de resultados, cada una enfocada en una dimensión del análisis:

\begin{itemize}
	\item \textbf{Operaciones por hora:} gráfica de líneas con marcadores que muestra la distribución horaria de operaciones exitosas por agente, revelando los picos de carga durante el servicio del mediodía y la noche.
	\item \textbf{Tasa de éxito por rol:} gráfica de barras con etiquetas de porcentaje que compara la fiabilidad operativa de cada agente durante las 8 horas.
	\item \textbf{Operaciones versus errores:} gráfica de barras agrupadas que contrasta el volumen de transacciones exitosas frente a los errores registrados por cada rol.
	\item \textbf{Errores por tipo:} gráfica de barras horizontales que desglosa los fallos por categoría y agente; se omite automáticamente si no se registraron errores durante la prueba.
	\item \textbf{Métodos de pago:} gráfica de torta con la distribución porcentual de los métodos de pago procesados por el agente Caja.
	\item \textbf{Línea de tiempo de actividad:} serie temporal con áreas sombreadas que muestra la actividad agregada en ventanas de 15 minutos, permitiendo identificar visualmente la evolución de la carga a lo largo de la jornada completa.
\end{itemize}

\paragraph{Etapa 4 — Exportación de datos y reporte de consola.} El script serializa cuatro archivos CSV derivados: el registro maestro consolidado con todos los eventos unificados, el resumen estadístico por agente, la actividad horaria desagregada y el catálogo detallado de fallos. Paralelamente, imprime en consola un reporte tabular estructurado que resume el periodo de ejecución, la duración aproximada, las métricas por rol y las estadísticas financieras de Caja, proporcionando una lectura inmediata de los resultados sin necesidad de abrir archivos adicionales.

\subsection{Análisis de Vulnerabilidades de Superficie con OWASP ZAP}

Como complemento a las pruebas unitarias de seguridad lógica, se integró un análisis dinámico de seguridad empleando la herramienta \textit{OWASP ZAP} (Zed Attack Proxy). El propósito de esta fase de evaluación es inspeccionar la superficie de ataque de la aplicación web expuesta en la red local, focalizándose en las debilidades a nivel de infraestructura HTTP y en la configuración de las defensas perimetrales.

Dado el diseño del sistema, donde el acceso a los módulos operativos requiere autenticación estricta y un segundo factor de hardware, el escaneo automatizado se concentra en el perímetro público inicial. Esto abarca la vista de inicio de sesión y los recursos estáticos disponibles sin credenciales previas, permitiendo auditar la robustez del servidor frente a peticiones anónimas. Las dimensiones de seguridad evaluadas mediante esta herramienta incluyen:

\begin{itemize}
	\item \textbf{Políticas de Seguridad de Contenido (CSP):} auditoría de las directivas en las respuestas HTTP (como las fuentes permitidas para la ejecución de \textit{scripts} y estilos), con el fin de evaluar el nivel de protección ante posibles ataques de inyección de código y secuencias de comandos entre sitios (XSS).
	\item \textbf{Cabeceras de Seguridad Complementarias:} verificación de la correcta implementación de defensas pasivas recomendadas, tales como la mitigación de ataques de superposición de interfaz visual (\textit{clickjacking}) y la prevención de vulnerabilidades por inferencia de tipos MIME.
	\item \textbf{Gestión de Sesión y Tokens Antifalsificación:} inspección de las propiedades de seguridad en las \textit{cookies} emitidas por el servidor (garantizando atributos de aislamiento estricto) y validación de la correcta emisión de los tokens criptográficos gestionados por ASP.NET Core para prevenir la falsificación de solicitudes entre sitios (CSRF).
\end{itemize}

\subsection{Implementación Piloto en Entorno Real}
\label{subsec:piloto}

Aunque no se contempló dentro de los objetivos iniciales del proyecto, se llevó a cabo
un despliegue operativo en un escenario comercial real: el establecimiento Gastro Uno 58 Bar.
El propósito de esta fase adicional fue validar el comportamiento del software fuera del
entorno de laboratorio y recolectar datos cualitativos sobre su nivel de adopción por parte
de usuarios sin formación técnica previa.

La prueba piloto abarcó un periodo de un mes de operación continua. El despliegue comenzó
con una jornada exclusiva de capacitación durante el día uno, en la cual se instruyó al
personal en el uso de las interfaces, el flujo de procesamiento de comandas y la autenticación
de doble factor mediante tarjetas NFC. La plantilla del restaurante estuvo conformada por cuatro usuarios operativos un administrador (quien asumió simultáneamente las funciones del módulo de Caja), dos meseras y una cocinera.

Debido a que el establecimiento contaba con un servicio activo de internet, el sistema operó
bajo las condiciones habituales de tráfico de una red comercial abierta, permitiendo contrastar
su estabilidad frente a las pruebas de intranet aislada previamente ejecutadas en el laboratorio.

Para evaluar el impacto de la herramienta, la metodología definió la recolección de resultados
a través de un cuestionario de satisfacción aplicado al personal al finalizar el mes de uso.
Este instrumento se estructuró en torno a seis dimensiones de evaluación: eficiencia operativa,
estabilidad técnica, usabilidad, reducción de errores, comunicación interna y control
administrativo y de seguridad. Las dimensiones de eficiencia operativa, estabilidad técnica,
usabilidad, reducción de errores y comunicación interna fueron evaluadas por la totalidad del
personal operativo, mientras que las dimensiones de control administrativo, doble factor de

Adicionalmente, el cuestionario incluyó una pregunta abierta orientada a identificar
funcionalidades que el personal desearía modificar o agregar, con el fin de documentar
retroalimentación cualitativa directa desde el contexto operativo real. 

Esta implementación en producción se utilizó también como modelo base para estructurar un
análisis de viabilidad económica. Dado que los componentes del módulo de seguridad por
hardware fueron aportados directamente por el autor del proyecto, el establecimiento no
incurrió en costo alguno durante el piloto. No obstante, se construyeron dos modelos de
referencia para representar el costo real que asumiría un tercero al adoptar la solución:
un escenario de adaptación, orientado a establecimientos que ya disponen de equipos de
cómputo y únicamente requieren adquirir el módulo de autenticación por hardware; y un
escenario de implementación desde cero, que contempla la adquisición de toda la
infraestructura necesaria para cada puesto operativo.
